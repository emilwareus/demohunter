Plan 01 defines the canonical narration request boundary. Plan 02 should key and persist cache entries from that exact structure:

```typescript
type NarrationRequest = {
  provider: "openai";
  model: string;
  voice: string;
  format: string;
  sampleRate: number;
  instructions: string;
  text: string;
};
```

The Phase 1 spec already recommends a content-addressed hash over provider/model/voice/instructions/format/sampleRate/normalized text/version. Use that as the cache identity basis, not ad hoc filenames or mutable request state.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| narration request -> cache identity | User-configured narration inputs decide whether assets are reused or regenerated |
| local filesystem -> cache metadata | On-disk audio and metadata can be missing, stale, or tampered with |
| maintenance commands -> cache root | Prune and clear can delete local data if their scope is wrong |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-02-01 | T | cache identity | mitigate | Hash only material request fields plus version in `cache-key.ts` so collisions do not arise from omitted voice/model/instructions/format/sampleRate inputs. |
| T-04-02-02 | D | corrupt cache entries | mitigate | Treat unreadable metadata, missing audio, and integrity mismatches as recoverable misses in `cache-store.ts` per D-06 instead of crashing the whole generation path. |
| T-04-02-03 | T | prune scope | mitigate | Keep maintenance helpers rooted to `.demohunter/cache` and conservative by default so prune only removes clearly invalid or obsolete entries per D-10. |
</threat_model>
