Phase 3 currently models narration this way:

```typescript
export type CollectedNarration = {
  kind: "narration";
  order: number;
  text: string;
  chapterTitle?: string;
  durationMs: number;
  event: NarrationRuntimeEvent;
};
```

Plan 03 should extend this into a real narration-segment contract with cached audio identity/timing that the output writer can serialize directly into subtitles:

```typescript
type NarrationSegment = {
  text: string;
  chapterTitle?: string;
  durationMs: number;
  audioPath: string;
  cacheKey: string;
};
```

Subtitles must be generated from these narration segments only.
</interfaces>

## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| pass-1 narration events -> cache/provider layer | User-authored narration text becomes external-provider and local-cache input |
| measured cache metadata -> replay timing | Pass-2 timing trusts the cached-audio duration recorded during pass 1 |
| narration segments -> subtitle files | User-visible caption output is derived from generator timeline data |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-04-03-01 | D | offline generation path | mitigate | Resolve narration through the cache service during pass 1 and only require `OPENAI_API_KEY` on uncached misses per D-02 and TTS-07/TTS-08. |
| T-04-03-02 | T | narration timing | mitigate | Use the measured cached-audio duration returned by the cache layer as the sole pass-1 timing source per D-07, with tests that forbid heuristic fallbacks. |
| T-04-03-03 | T | subtitle derivation | mitigate | Serialize SRT/VTT exclusively from narration segments and add tests that reject chapter/step/highlight events as subtitle inputs per D-08 and OUT-03. |
</threat_model>
