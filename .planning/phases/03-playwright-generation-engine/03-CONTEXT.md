# Phase 3: Playwright Generation Engine - Context

**Gathered:** 2026-04-10
**Status:** Ready for planning

<domain>
## Phase Boundary

Execute tours in two passes, record the scripted demo, and write the baseline local artifact set to `.demohunter/`. This phase covers the real generation engine, recorded-pass strictness, overlays, and baseline local output behavior; it does not add OpenAI-backed narration synthesis, cache commands, subtitle generation, manifest schema finalization, or cloud behavior.

</domain>

<decisions>
## Implementation Decisions

### App readiness and Playwright responsibility
- **D-01:** DemoHunter should not add extra app-readiness automation beyond normal Playwright behavior.
- **D-02:** Users remain responsible for having their app running, reachable, and set up correctly for Playwright before `demohunter generate` starts.
- **D-03:** Phase 3 should lean into direct Playwright behavior rather than trying to abstract or auto-heal app startup and bootstrap concerns.

### Recording overlays
- **D-04:** When `record.showActions` and `record.showChapters` are enabled, overlays should be minimal and brief rather than persistent or heavy-handed.
- **D-05:** Overlay behavior should feel instructional without turning the output into a built-in player or presentation framework.

### Output behavior
- **D-06:** `mp4` should be the default recorded artifact in this phase.
- **D-07:** `webm` should be opt-in rather than generated by default.

### Pass-2 strictness
- **D-08:** The recorded pass should be strict and fail clearly if it diverges from the pass-1 timeline or expected page state.
- **D-09:** Best-effort output is not acceptable when determinism breaks; preserving the integrity of the generation contract is more important than emitting a flawed artifact.

### the agent's Discretion
- Exact pass-1 and pass-2 internal data structures, replay mechanics, and error taxonomy are open to research and planning as long as the two-pass contract stays explicit.
- Specific overlay visuals, timing, and rendering implementation are open to planning provided they stay minimal and brief.
- The exact baseline output set for this phase can be scoped within the roadmap and requirements boundaries, provided it remains consistent with the later portable output contract.

</decisions>

<code_context>
## Existing Code Insights

### Reusable Assets
- `packages/generator-playwright/src/smoke-generate.ts`: already launches Playwright, executes a tour against a shared runtime, validates tour IDs, and writes a smoke artifact under the configured output root.
- `packages/generator-playwright/src/runtime/create-smoke-tour-runtime.ts`: already emits structured runtime events for chapters, steps, narration, snapshots, highlights, stability waits, and assertions.
- `packages/cli/src/commands/generate.ts`: already loads config, imports tours, validates the tour contract, and routes execution into the generator package.
- `packages/sdk/src/runtime-types.ts`: already defines the Phase 2 runtime helper contract that Phase 3 can observe and replay.

### Established Patterns
- DemoHunter stays a thin wrapper on top of Playwright rather than replacing Playwright semantics or taking over app bootstrap.
- The repo already treats source and built CLI flows as product contracts and validates both with e2e coverage.
- The current smoke generator preserves strict local output-root behavior and rejects unsafe tour IDs, which Phase 3 should carry forward into the real engine.

### Integration Points
- Phase 3 will primarily expand `packages/generator-playwright` from a smoke runner into a true two-pass executor/recorder.
- The runtime event stream introduced in Phase 2 is the obvious input for pass-1 timeline capture and pass-2 replay behavior.
- `demohunter generate` in the CLI should continue to route through the generator package without adding app-specific readiness automation.

</code_context>

<specifics>
## Specific Ideas

- The generation engine should lean into Playwright instead of trying to automate app startup or patch missing app readiness.
- Minimal and brief overlays are preferred over persistent or visually dominant chapter/action callouts.
- Strict recorded-pass failure is preferred over best-effort output when the replay diverges from the captured timeline.

</specifics>

<deferred>
## Deferred Ideas

- TTS-backed narration synthesis, cache policy, and offline regeneration rules — Phase 4.
- Subtitle generation and narration-duration timing from real audio — Phase 4.
- Manifest schema, checksums, and full portable output finalization — Phase 5.
- Rich playback-style overlays or presentation-layer polish beyond brief recording annotations — out of scope for this phase.

</deferred>
