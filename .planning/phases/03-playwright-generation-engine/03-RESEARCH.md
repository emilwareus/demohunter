# Phase 3: Playwright Generation Engine - Research

**Researched:** 2026-04-10
**Domain:** Two-pass Playwright-based demo generation and recording
**Confidence:** HIGH

<user_constraints>
## User Constraints (from CONTEXT.md)

### Locked Decisions
- DemoHunter should not add extra app-readiness automation beyond normal Playwright behavior.
- Users remain responsible for having their app running, reachable, and set up correctly for Playwright before `demohunter generate` starts.
- Phase 3 should lean into direct Playwright behavior rather than trying to abstract or auto-heal app startup and bootstrap concerns.
- When `record.showActions` and `record.showChapters` are enabled, overlays should be minimal and brief rather than persistent or heavy-handed.
- Overlay behavior should feel instructional without turning the output into a built-in player or presentation framework.
- `mp4` should be the default recorded artifact in this phase.
- `webm` should be opt-in rather than generated by default.
- The recorded pass should be strict and fail clearly if it diverges from the pass-1 timeline or expected page state.
- Best-effort output is not acceptable when determinism breaks; preserving the integrity of the generation contract is more important than emitting a flawed artifact.

### the agent's Discretion
- Exact pass-1 and pass-2 internal data structures, replay mechanics, and error taxonomy are open to research and planning as long as the two-pass contract stays explicit.
- Specific overlay visuals, timing, and rendering implementation are open to planning provided they stay minimal and brief.
- The exact baseline output set for this phase can be scoped within the roadmap and requirements boundaries, provided it remains consistent with the later portable output contract.

### Deferred Ideas (OUT OF SCOPE)
- TTS-backed narration synthesis, cache policy, and offline regeneration rules.
- Subtitle generation and narration-duration timing from real audio.
- Manifest schema, checksums, and full portable output finalization.
- Rich playback-style overlays or presentation-layer polish beyond brief recording annotations.
</user_constraints>

<research_summary>
## Summary

Phase 3 should build the real generator around Playwright’s current first-class screencast support rather than stretching the Phase 1 smoke runner or treating Playwright video recording as a test-runner feature. Official Playwright 1.59 docs now expose `page.screencast.start()`, `stop()`, and `showActions()` directly, which aligns with the product spec’s explicit generator responsibilities and the project’s minimum Playwright version. That makes the correct architectural move a two-pass executor that records and replays against the existing Phase 2 runtime event stream, not a generic `recordVideo` wrapper or a custom automation layer.

The repo is already set up for this: Phase 2 introduced typed runtime events for chapter, step, narration, snapshot, highlight, stability waits, and assertions, and the current smoke runner already validates tours, enforces filesystem-safe IDs, and proves source plus built consumer flows. Phase 3 therefore should focus on turning those runtime events into a deterministic pass-1 timeline and a strict pass-2 replay/recording engine. That keeps the system Playwright-native while preserving the roadmap’s insistence on determinism and local-first artifact writing.

The most important planning implication is artifact scope. The product docs describe video, subtitles, chapters, and manifest as part of the broader OSS contract, but this roadmap phase is specifically the “baseline local artifact set.” Because subtitles, narration-duration timing, and manifest schema are deferred to Phases 4 and 5, Phase 3 should deliver the real recorder, strict two-pass flow, and baseline video/chapter-oriented output scaffolding without prematurely finalizing later-phase output schema responsibilities.

**Primary recommendation:** Plan Phase 3 around a strict two-pass engine that uses Phase 2 runtime events as the single source of truth, records via Playwright’s `page.screencast` API, keeps overlays brief, and writes only the baseline video-oriented artifacts needed before TTS/cache/schema phases land.
</research_summary>

<architecture_patterns>
## Architecture Patterns

### Recommended Project Structure
```text
packages/generator-playwright/src/
├── execute/
│   ├── collect-timeline.ts       # pass-1 event/timing capture
│   ├── replay-timeline.ts        # pass-2 strict replay logic
│   └── generator-types.ts        # timeline and output contracts
├── overlays/
│   ├── action-overlay.ts         # brief action annotations
│   └── chapter-overlay.ts        # brief chapter markers
├── record/
│   ├── screencast.ts             # Playwright screencast integration
│   └── mux-video.ts              # mp4 default / webm opt-in conversion
└── smoke-generate.ts             # upgraded into real generate orchestration or delegated wrapper
```

### Pattern 1: Pass-1 timeline as the sole replay contract
**What:** Run the tour once to collect timeline events and timing metadata, then feed that exact timeline into pass 2.
**When to use:** Always for Phase 3 generation.
**Example:**
```typescript
const timeline = await collectTimeline({ page, tour, runtimeEvents });
await replayTimeline({ page, timeline, strict: true, holdPaddingMs });
```

### Pattern 2: Strict replay with explicit divergence errors
**What:** Treat pass-2 mismatches as generator errors rather than soft warnings.
**When to use:** Whenever a step, narration anchor, or expected state no longer matches pass 1.
**Example:**
```typescript
if (!matchesExpectedStep(actualEvent, expectedEvent)) {
  throw new Error(
    `Recorded pass diverged at step "${expectedEvent.title}" in chapter "${expectedEvent.chapterTitle ?? "unscoped"}"`,
  );
}
```

### Pattern 3: Playwright-native recording control
**What:** Use `page.screencast` to start/stop recorded capture and Playwright’s own action annotation support when it matches the desired output.
**When to use:** In pass 2 after the replay context is ready.
**Example:**
```typescript
await page.screencast.start({ path: tempWebmPath, size: viewport });
if (config.record.showActions) {
  await page.screencast.showActions({ duration: 500, position: "top-right" });
}
// replay timeline
await page.screencast.stop();
```

### Anti-Patterns to Avoid
- **Trying to manage app startup:** Generator retries, custom readiness daemons, or implicit `wait-for-server` logic contradict locked decisions and thin-wrapper positioning.
- **Timing via heuristics in Phase 3:** The event model should be explicit and deterministic, not based on arbitrary sleep guesses that later TTS phases have to undo.
- **Overcommitting later artifacts too early:** Subtitles, final manifest schema, and full portable output should not be silently finalized in this phase.
</architecture_patterns>

<dont_hand_roll>
## Don't Hand-Roll

Problems that look simple but have existing solutions:

| Problem | Don't Build | Use Instead | Why |
|---------|-------------|-------------|-----|
| Browser video capture | Custom frame polling / screenshot stitching | Playwright `page.screencast` or `recordVideo` | Native APIs handle browser-integrated capture more reliably |
| Action decorations | Entirely custom overlay engine for interacted elements | Playwright `screencast.showActions()` where it fits | Built-in annotations reduce complexity for the action-overlay part |
| File URL generation in tests or temp repos | String-concatenated `file://` paths | `pathToFileURL(...).href` | Cross-platform correctness is already a proven source of bugs in this repo |
| Output path safety | Ad hoc path joins with raw user values | Existing tour-id slug validation and explicit path guards | Prevents `.demohunter/` layout drift and path traversal bugs |

**Key insight:** let Playwright own browser behavior and capture, and let DemoHunter own orchestration, determinism, and artifact contracts.
</dont_hand_roll>

<common_pitfalls>
## Common Pitfalls

### Pitfall 1: Confusing “two pass” with “run twice and hope”
**What goes wrong:** Pass 2 simply reruns the tour without a first-class timeline contract.
**Why it happens:** The smoke runner already executes a tour, so it is tempting to bolt recording onto that path.
**How to avoid:** Make pass-1 outputs explicit and feed them into pass 2 as structured replay data.
**Warning signs:** Plans talk about “run once silently, run again with recording” without naming a concrete timeline artifact.

### Pitfall 2: Using the wrong Playwright recording primitive
**What goes wrong:** The implementation leans on test-oriented `recordVideo` behavior when the product needs explicit recording control and overlays.
**Why it happens:** `recordVideo` is familiar and broadly documented.
**How to avoid:** Prefer `page.screencast` for Phase 3 because the official Playwright 1.59 API now exposes explicit recording and action-annotation control there.
**Warning signs:** Plans depend entirely on `page.video()` artifacts and never mention `page.screencast`.

### Pitfall 3: Letting cleanup or recording shutdown hide the real failure
**What goes wrong:** Browser-context close or recording-stop errors overwrite the primary replay failure.
**Why it happens:** Cleanup is often placed in broad `finally` blocks without preserving the main error.
**How to avoid:** Preserve the primary pass failure and only surface cleanup errors when no primary error exists.
**Warning signs:** `finally` blocks throw close/stop errors unconditionally after a failed replay.
</common_pitfalls>

<open_questions>
## Open Questions (RESOLVED)

1. **Should Phase 3 use `recordVideo` or `page.screencast` as the primary recording primitive?**
   - Resolved answer: use `page.screencast` as the primary Phase 3 recording primitive.
   - Why: official Playwright 1.59 docs now provide explicit screencast start/stop and action annotation support, which matches the product spec and the need for strict pass-2 control.
   - Planning implication: include a dedicated recording integration task rather than treating video as incidental context output.

2. **How much of the final portable artifact contract belongs in Phase 3?**
   - Resolved answer: Phase 3 should deliver baseline video-oriented local output writing only, without finalizing subtitle, manifest-schema, or full portability semantics that are explicitly deferred to later phases.
   - Why: roadmap phases 4 and 5 already own those responsibilities.
   - Planning implication: tasks should write only the baseline local artifacts needed to prove the generator engine, while leaving schema and subtitle finalization to later phases.

3. **Where should strict replay behavior live?**
   - Resolved answer: strictness belongs inside the generator/replay layer, not in authored tour code or the SDK.
   - Why: authors should keep writing Playwright-like tours, while the generator owns determinism guarantees.
   - Planning implication: plans should include explicit replay-validation/error-handling work inside `packages/generator-playwright`.
</open_questions>
